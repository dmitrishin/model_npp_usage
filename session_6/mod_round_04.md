## Наиболее значимые доработки идей

### 1. Claim-менеджмент (R2-1-01, R2-3-09, R1-7-07)

**Критики:** C-1-06, C-2-09, C-3-09, C-4-16, C-8-05  
**Доработка:** Модель переориентирована с «доказательства в суде» на внутренний инструмент подготовки claim, обоснования переговорной позиции и досудебного урегулирования. Для арбитража — помощь в подготовке экспертного заключения, но не самостоятельное доказательство.

### 2. Интеграция с BIM/PIM (R2-3-02)

**Критики:** C-1-10, C-2-10, C-3-02, C-5-03, C-6-03  
**Доработка:** Переход к поэтапной гибридной интеграции. Этап 1: полуавтоматическая синхронизация только по критическому пути (еженедельный импорт с валидацией). Снято утверждение об «абсолютно точном прогнозе без участия человека».

### 3. Сквозной PERT→GERT (R1-2-01, R1-3-06)

**Критики:** C-2-01, C-3-03, C-4-07  
**Доработка:** Введена иерархическая двухуровневая архитектура. Подготовительный период — GERT с акцентом на внешние события. Основной — детальная PERT/Monte Carlo + ML. Связь через интерфейсные события. Rolling wave для управляемой гранулярности.

### 4. ML на данных Росатома (R1-1-17, R1-3-04, R1-4-14, R2-3-08)

**Критики:** C-1-01, C-7-02  
**Доработка:** Обучение на уровне типовых работ (тысячи операций), а не целых проектов. Иерархическая байесовская модель. Для головных блоков — упрощённые модели. Обязательный корпоративный стандарт сбора данных как предусловие. Модель — стимул для цифровизации учёта.

### 5. TGN реконфигурация графа (R2-7-05)

**Критики:** C-1-04, C-5-05  
**Доработка:** TGN переведена из режима «генератор решений» в режим «интеллектуальный селектор» из заранее заложенной технологами библиотеки допустимых альтернатив. Обязательная инженерная экспертиза каждого варианта.

### 6. Устойчивость цепочки поставок (R2-5-06)

**Критики:** C-2-08, C-4-19, C-6-05, C-7-04  
**Доработка:** Двухуровневая фильтрация сценариев (top-10 risk drivers). ОДЦИ ядерного острова исключены из модели альтернативных поставщиков (монопольный рынок). Для геополитических рисков — интервальные вероятности или сценарный анализ без вероятностей.

### 7. Портфельная оптимизация (R2-3-01, R2-8-02)

**Критики:** C-3-01, C-4-20, C-7-10, C-8-06  
**Доработка:** Федеративная рекомендательная система вместо централизованной. Право вето у руководителя проекта. Пилот на 2–3 проектах. GNN — как калькулятор последствий для диалога, а не «чёрный ящик».

### 8. Шаг ввода энергоблоков (R1-1-09, R1-4-13, R1-1-15)

**Критики:** C-1-12, C-8-09  
**Доработка:** Применение на предконтрактной стадии (когда решение ещё не зафиксировано). При уже определённом шаге — оценка реалистичности и обоснование ресурсной потребности.

### 9. Размывание фокуса модели (Кластер K-18 / K-19)

**Критика:** C-1-11  
**Доработка:** Поддержано 4 участниками (У1, У3, У7, У8). Каждый новый атрибут должен пройти формальный порог: (а) лежит на критическом пути, (б) влияние количественно значимо, (в) управляем на уровне проекта. Фокус модели — T, C, R, Q. Непрофильные факторы — внешние ограничения, а не оптимизируемые переменные.

## Ключевые сквозные выводы Раунда 4

**1. Качество данных как предусловие (консенсус)**  
Несколько участников (У1, У2, У4, У6, У8) сошлись на том, что внедрение ML-компонентов модели невозможно без предварительного создания корпоративного стандарта сбора и верификации данных. Модель рассматривается как стимул для цифровизации учёта, а не как инструмент, требующий готовых данных.

**2. Иерархическая архитектура модели (консенсус)**  
Участники 1, 2, 3, 4 сошлись на необходимости двухуровневой архитектуры (подготовительный + основной период) вместо единой «плоской» сети. Связь через интерфейсные события.

**3. Разделение рисков на классы (консенсус)**  
Участники 4, 5, 6 предложили разделять риски: частые с известными распределениями (Monte Carlo) vs. редкие/уникальные без вероятностей (сценарный анализ, критерий Вальда).

**4. Регулятор — не «игрок» (полный консенсус)**  
Все участники, высказавшиеся по теме (6 из 8), поддержали снятие идеи R2-3-04. Регулятор моделируется как процесс с фиксированными процедурами.

**5. Фокус модели на T, C, R, Q (широкий консенсус)**  
Поддержано ограничение пространства оптимизации четырьмя основными измерениями статьи. Периферийные факторы — как внешние ограничения.
