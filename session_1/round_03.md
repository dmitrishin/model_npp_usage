Раунд 3 — Критика

C-1-01
ID идеи: R2-4-04
Слабое место: Reinforcement Learning для диспетчеризации ресурсов на стройплощадке имеет фундаментальную проблему определения пространства действий и функции вознаграждения. Реальное пространство состояний стройплощадки АЭС (тысячи работ, сотни бригад, десятки объектов, погодные условия, состояние фронтов) экспоненциально велико. RL-агент требует миллионов эпизодов для обучения — невозможно получить такой объём данных ни из истории (десятки проектов, не миллионы), ни из симуляций (требуется высокоточная среда, которая сама по себе является нерешённой задачей). Функция вознаграждения, включающая безопасность, качество, сроки и стоимость одновременно, порождает проблему мультицелевого RL с неочевидным балансом весов. Кроме того, руководство проекта АЭС не примет «чёрный ящик» RL-агента для диспетчерских решений без объяснимости — в отрасли, где каждое решение может быть предметом аудита и расследования.

C-1-02
ID идеи: R2-3-04
Слабое место: Physics-Informed Neural Networks (PINN) для моделирования физических процессов (набор прочности бетона, тепловые напряжения при сварке ГЦТ) имеют серьёзное ограничение применимости в контексте данной модели. Модель Дмитришина оперирует на уровне длительностей работ (дни, недели), а физические процессы типа набора прочности или тепловых напряжений происходят на масштабе часов и описываются хорошо известными инженерными моделями (формулы Боломея-Скрамтаева, FEM-расчёты напряжений). Использование PINN вместо этих устоявшихся моделей не даёт преимущества — оно усложняет систему, добавляя слой нейронных сетей поверх задач, которые решаются аналитически. Реальная длительность бетонных работ определяется не физикой набора прочности, а организацией (подача бетона, армирование, опалубка, ожидание фронта), что PINN не моделирует.

C-1-03
ID идеи: R2-3-07
Слабое место: Federated learning между проектами портфеля Росатома предполагает достаточную гомогенность данных между площадками. На практике проекты радикально различаются по: типу проекта (АЭС-2006, ВВЭР-ТОИ, экспортные), регуляторной среде (Россия, Турция, Египет, Бангладеш), генподрядчику и организационной структуре, системам учёта и форматам данных. Federated learning эффективен, когда участники решают сходную задачу на сходных данных (классический пример — мобильные клавиатуры). Для гетерогенного портфеля с 3–5 проектами одного типа объём данных каждого участника слишком мал для обучения локальной модели, а различия между участниками слишком велики для полезной агрегации градиентов. Более реалистичен простой transfer learning с ручной калибровкой, что уже предложено в R1-2-12.

C-1-04
ID идеи: R2-8-05
Слабое место: Сентимент-анализ локальных СМИ и социальных сетей для корректировки вероятностей GERT-модели имеет проблему калибровки: какое именно изменение тональности на сколько процентов меняет вероятность ветвления «отклонение заявки на слушаниях»? Без исторических данных о связи сентимента и регуляторных решений (которых для АЭС практически не существует — строится 1–2 станции в стране) калибровка будет произвольной. Кроме того, в странах-заказчиках экспортных АЭС (Египет, Бангладеш, Турция) информационное пространство часто контролируется государством, социальные сети — не индикатор реального общественного мнения, а протесты (если возникают) определяются политическими, а не информационными факторами. Идея имплицитно предполагает западную модель общественного участия, которая не соответствует реальности основных экспортных рынков Росатома.

C-1-05
ID идеи: R1-3-01
Слабое место: Построение единого многомерного предиктивного event-time графа как цифрового двойника всего проекта от подготовительного периода до этапа Г — идея масштабная, но обоснование заявляет «повышение вероятности соблюдения сроков на 15–25%» без какой-либо эмпирической основы. Такая количественная оценка создаёт ложное впечатление обоснованности. Реальная проблема: единый граф проекта АЭС содержит 50 000–100 000 работ. Многомерное пространство атрибутов (время, стоимость, риск, качество, ресурсы) на таком графе создаёт вычислительную задачу, для которой не продемонстрирована трактабельность. Декларируется непрерывная адаптация параметров по фактическим данным, но не обсуждается, как обеспечить качество и полноту этих данных в реальном времени на стройплощадке с тысячами рабочих и десятками подрядчиков.

C-1-06
ID идеи: R2-2-02
Слабое место: Байесовский контур оценки «истинного факта» выполнения работ — концептуально привлекательная идея, но содержит скрытое допущение о доступности множественных независимых сигналов о состоянии каждого узла (акты, журналы, протоколы НК, статус коллизий, готовность помещений и т.д.). На практике для большинства работ доступен 1–2 источника, и они коррелированы (один человек заполняет и акт, и журнал). Байесовское обновление при коррелированных источниках с малым числом сигналов не даёт статистически значимого «истинного факта» — оно даёт ещё одну оценку с собственной неопределённостью. Кроме того, обучение ML-модуля «определять достоверность источников» требует размеченных данных о случаях «бумажного прогресса» — крайне чувствительной информации, которую организации не склонны формализовать.

C-1-07
ID идеи: R2-1-07
Слабое место: Многоагентное моделирование организационных взаимодействий (заказчик, генподрядчик, субподрядчики, проектировщик, регулятор, поставщики) требует формализации функций полезности, информационных горизонтов и стратегий каждого агента. Для этого необходимо знать внутренние стимулы и ограничения каждой организации, которые составляют коммерческую тайну. Генподрядчик не раскроет свою реальную маржу и стратегию распределения ресурсов между проектами; субподрядчик не раскроет свои альтернативные контракты. Без достоверных функций полезности многоагентная модель будет оперировать предположениями о поведении участников, что может давать результаты хуже, чем простая экспертная оценка организационных рисков. Кроме того, сама публикация результатов такого моделирования (где один участник назван «нерациональным агентом с запаздыванием реакции 3 недели») может разрушить рабочие отношения.

C-1-08
ID идеи: R2-3-01
Слабое место: Distributionally robust optimization (DRO) с ambiguity set вокруг эмпирических распределений — математически строгий подход, но его практическая применимость к проекту АЭС ограничена. DRO решает задачу «worst-case distribution within ambiguity set», что для Wasserstein ambiguity set приводит к решению задачи semi-infinite programming. На графе из десятков тысяч работ с многомерными атрибутами вычислительная сложность DRO может оказаться непреодолимой. Более того, DRO имеет тенденцию к чрезмерному консерватизму: оптимизация под наихудшее возможное распределение даёт планы, которые «страхуют от всего», но при этом существенно дороже и длиннее среднего оптимума. Для проекта АЭС с директивными сроками и бюджетом гипер-консервативный план может быть неприемлем заказчику.

C-1-09
ID идеи: R2-3-06
Слабое место: Closed-loop система с IoT/компьютерным зрением для мгновенного байесовского обновления параметров графа имеет критическое ограничение в контексте ядерного строительства: территория строительства АЭС — режимный объект с жёсткими требованиями физзащиты, ограничениями на фото-/видеосъёмку и электронные устройства (особенно с передачей данных). Развёртывание сети камер и IoT-датчиков на объекте, где действуют требования ОПБ-88/97 и правила физзащиты, потребует отдельного согласования с регулятором. Компьютерное зрение для оценки строительного прогресса (например, процент заполнения опалубки) работает в лабораторных условиях, но на реальной площадке с пылью, вибрацией, переменным освещением, лесами и укрытиями точность существенно падает. Заявка на «реальное время» также неоправданна: пересчёт Monte Carlo на графе из тысяч работ после каждого IoT-события физически невозможен в реальном времени.

C-1-10
ID идеи: R1-1-16
Слабое место: Применение графовых нейронных сетей (GNN) для выявления скрытых зависимостей между работами предполагает, что такие зависимости существуют, не отражены в сетевом графике и могут быть выявлены из данных. Первое допущение верно (неформальные зависимости есть), но второе проблематично: скрытые зависимости не оставляют следов в структурированных данных проектного учёта. Если бригада A задержалась, потому что бригада B заняла кран (неформальная зависимость), в проектной системе это отразится как задержка работы A с причиной «ожидание техники», а не как связь A→B. GNN для обнаружения таких связей требует данных уровня «кто где в какой момент» — данных, которые проектные системы Росатома (СПФ/ИСУП) не фиксируют с необходимой детальностью. Без таких данных GNN будет находить корреляции в шуме.

C-1-11
ID идеи: R2-6-02
Слабое место: Инвертированная итеративная оптимизация для «растяжения» субкритических путей под годовые лимиты экспортного кредита — оригинальная идея, но содержит скрытый конфликт интересов. Целенаправленное замедление некритических работ для выравнивания освоения по траншам может создать ситуацию, когда работы, имевшие запас по времени, его лишаются. Если после «растягивания» на некритическом пути возникнет незапланированная задержка, он может стать новым критическим путём — а запас уже израсходован на финансовую оптимизацию. Кроме того, генподрядчику «замедлять» работы — контринтуитивно и может привести к демобилизации бригад и техники, которую потом сложно мобилизовать обратно. Финансовая логика здесь конфликтует с производственной.

C-1-12
ID идеи: R2-6-03
Слабое место: Автоматизированный математический фильтр заявок на изменения (Change Request) через графовые автоэнкодеры с автоматическим отклонением заявок по критерию Вальда содержит серьёзную управленческую проблему. Изменения проекта АЭС бывают обязательными (требования регулятора, дефекты оборудования, результаты анализа безопасности) — их нельзя «математически бракировать», даже если они разрушают структуру графа. Система, которая автоматически отклоняет Change Request, может заблокировать критически важные изменения по безопасности. Кроме того, автоэнкодер, обученный на «эталонной» топологии одного проекта, имеет выборку из одного примера — это не обучение, а запоминание. Ошибка реконструкции будет реагировать на любое изменение, а не только на «опасное», порождая ложные срабатывания.

C-1-13
ID идеи: R1-5-05
Слабое место: GNN для обнаружения скрытых зависимостей в многомерном пространстве событий имеет проблему обучающей выборки. Каждый проект АЭС — единичный граф из десятков тысяч узлов. Для обучения GNN нужны множественные графы со схожей структурой, но различными реализациями (как множество молекул в drug discovery). У Росатома за всю историю — десятки блоков, причём с разными типами реакторов, разными стандартами учёта и разными системами классификации работ. Обучение GNN на графе одного проекта для предсказания поведения другого потребует строгой гомологии структуры графов, которой между проектами ВВЭР-1000, ВВЭР-1200 и ВВЭР-ТОИ не существует.

C-1-14
ID идеи: R2-9-03
Слабое место: Вероятностная модель общественных слушаний и социального противодействия с применением критерия Вальда для выбора стратегии коммуникации переоценивает формализуемость общественных процессов. Критерий Вальда требует определения множества состояний природы S и полезности каждой стратегии в каждом состоянии. Для общественных слушаний состояния природы — это комбинации политических, социальных, экономических и культурных факторов, которые невозможно свести к дискретному множеству. «Вероятности ветвлений на основе социологических данных региона» — экспертные оценки, замаскированные под данные. Модель создаёт иллюзию количественной обоснованности для решений, которые по своей природе являются политическими и переговорными.

C-1-15
ID идеи: R1-1-25
Слабое место: СППР, реализующая полный цикл модели (10 модулей: DAG, PERT, GERT, Monte Carlo, критические пути, локальная и итеративная оптимизация, многомерный граф, ML, многокритериальная оптимизация, обнаружение аномалий, визуализация) — это не идея применения модели, а описание программного продукта. Как идея она не отвечает на ключевой вопрос: к какой конкретной проблеме сооружения АЭС это применяется? «Ко всем» — это не ответ, а уклонение от приоритизации. Разработка такой системы — проект на годы с бюджетом в сотни миллионов рублей, конкурирующий с существующими корпоративными системами (Primavera, Spider Project, ИСУП). Идея не обсуждает интеграцию с ними, стратегию внедрения, change management и сопротивление пользователей — а именно эти факторы определяют, будет ли система использоваться.

C-1-16
ID идеи: R2-1-13
Слабое место: Temporal Graph Networks (TGN) для прогнозирования «волн» каскадных задержек — привлекательная аналогия с эпидемиологическими моделями, но обучение TGN требует временных рядов изменений графа по множеству проектов. Один проект — один временной ряд. Для TGN нужны сотни–тысячи реализаций, где задержки распространялись по-разному. Данных такого масштаба не существует: проекты Росатома ведутся в разных системах учёта, гранулярность записи прогресса различается (от ежедневного до ежемесячного), а ретроспективная реконструкция динамики графа из архивов P6/Spider Project — отдельная трудоёмкая задача. Без достаточного объёма обучающих данных TGN будет переобучаться на единственном доступном проекте.

C-1-17
ID идеи: R2-3-02
Слабое место: Value of Information (VoI) анализ для приоритизации предпроектных исследований имеет проблему рекурсии: для расчёта VoI нужно знать, как результаты исследования изменят распределения параметров в модели, но именно это и является предметом неопределённости. Чтобы оценить «ожидаемое снижение неопределённости» от дополнительных геотехнических изысканий, нужно знать, какой результат они дадут — а если бы мы это знали, изыскания были бы не нужны. VoI работает, когда можно параметрически описать возможные результаты исследования и их вероятности — для геотехнических условий (скальные/полускальные/глинистые грунты, уровень грунтовых вод) это возможно лишь грубо, и результат VoI будет определяться качеством этой грубой оценки, а не «количественной приоритизацией».

C-1-18
ID идеи: R2-3-03
Слабое место: Multi-agent GERT-модель с mechanism design для автоматического проектирования оптимальных контрактов предполагает, что поведение участников (генподрядчик, поставщики, регулятор) можно описать рациональными функциями полезности, что применим revelation principle, и что центральный оптимизатор может найти incentive-compatible механизм. Но контрактные отношения в ядерном строительстве определяются не только экономическими стимулами: регулятор максимизирует безопасность (не полезность), государственный заказчик подчиняется политическим ограничениям, генподрядчик в структуре Росатома — не независимый агент. Mechanism design требует private information revelation — участники должны раскрыть свои «типы», чего в реальных переговорах не происходит. Идея предлагает решение задачи теории контрактов уровня нобелевской премии (Hurwicz, Myerson, Maskin) как прикладной инструмент, не обсуждая условия применимости.

C-1-19
ID идеи: R2-5-08
Слабое место: Оптимизация графика обращения с РАО, образующимися в ходе строительства — идея крайне узкая. В ходе строительства АЭС (до загрузки топлива) радиоактивных отходов образуется минимально: только источники для радиографического контроля (Ir-192, Se-75), которые являются закрытыми источниками и возвращаются поставщику. Объём логистики РАО на этапе строительства настолько мал, что его оптимизация не оправдывает построения вероятностной модели. Это проблема, решаемая стандартными организационными процедурами.

C-1-20
ID идеи: R2-5-16
Слабое место: Оптимизация графика геодезического мониторинга осадок — задача, слабо связанная с моделью Дмитришина. Мониторинг осадок регулируется нормативными документами (СП 22.13330, СП 126.13330), которые предписывают частоту наблюдений в зависимости от этапа строительства. Вероятностная оптимизация частоты измерений не нужна — она задана нормами. Сам мониторинг не находится на критическом пути и не является фактором неопределённости проекта. Синхронизация с этапами бетонирования тривиальна и не требует Monte Carlo, GERT или ML.

C-1-21
ID идеи: R2-5-19
Слабое место: Оптимизация графика противопожарных мероприятий и испытаний систем пожаротушения — аналогично R2-5-16, это нормативно регулируемая область (ФЗ-123, НП-044-18), где последовательность и объём мероприятий определены требованиями безопасности, а не оптимизацией. Нельзя «оптимизировать» время проведения испытания системы пожаротушения, если нормативный документ предписывает провести его до начала определённых работ. Модель Дмитришина наиболее ценна для задач с реальной неопределённостью и свободой выбора, а не для формально предписанных процедур.

C-1-22
ID идеи: R2-5-20
Слабое место: Оптимизация страхового покрытия проекта на основе Monte Carlo и QRA — задача, лежащая за пределами компетенции модели управления проектом. Страховые премии определяются не внутренней вероятностной моделью заказчика, а оценкой страховщика (которая основана на его собственном актуарном анализе). Предоставление страховщику внутренней модели рисков может привести к обратному эффекту: страховщик увидит риски, которые раньше не учитывал, и повысит премию. Кроме того, оптимизация «франшиз и лимитов» — задача финансового менеджмента, не требующая GERT, Monte Carlo и ML.

C-1-23
ID идеи: R1-7-01 — R1-7-07
Слабое место: Серия из 7 идей Участника 7 в раунде 1 (оптимизация графика подготовительного периода, прогнозирование стоимости, оптимизация поставок, координация подрядчиков, управление рисками, анализ данных, улучшение качества) сформулирована на уровне общих деклараций без указания конкретных механизмов модели. Например, R1-7-01 «модель может быть использована для более точного прогнозирования сроков» — это описание цели, а не идеи применения. Какой именно компонент модели (PERT? GERT? Monte Carlo? ML?) применяется к какому конкретному объекту (лицензирование? изыскания? РД?) — не указано. Все 7 идей фактически повторяют формулировку задачи мозгового штурма, а не предлагают решения.

C-1-24
ID идеи: R2-9-06
Слабое место: ML-моделирование немонотонной производительности бригад при переработках — реальная проблема, но предлагаемое решение (нейронные сети/XGBoost для прогнозирования производительности как функции нагрузки) наталкивается на ограничение данных. Для обучения модели нужны данные: (фактическая выработка бригады за смену, количество часов работы за последние N дней, тип работ, условия). Такие данные системно не собираются: фиксируются акты выполненных работ (агрегированные за отчётный период), а не посменная производительность. Даже если начать сбор — выборка с одного проекта будет содержать десятки, а не тысячи наблюдений для каждого типа работ, чего недостаточно для обучения нелинейной модели.

C-1-25
ID идеи: R2-3-05
Слабое место: Иерархическая многоуровневая оптимизация (стратегический → тактический → операционный уровень) с итеративным пересчётом снизу вверх — классическая схема декомпозиции, но содержит скрытую проблему частоты итераций. Стратегический уровень (контрактная стратегия, структура финансирования) фиксируется на годы. Тактический (локальная оптимизация сегментов ТМР) — на месяцы. Операционный (ежедневное перепланирование) — на часы. Пересчёт «снизу вверх» означает, что ежедневные отклонения должны влиять на стратегические решения — но стратегические решения (контракт подписан, финансирование выделено) не могут меняться ежедневно. Реальная полезность обратной связи ограничена тактическим уровнем; стратегический уровень может быть пересмотрен только при драматических отклонениях, что не требует автоматического пересчёта.

C-1-26
ID идеи: R2-2-05
Слабое место: Brownfield-модель с окнами ППР действующих блоков — очень практичная идея, но содержит скрытое допущение о предсказуемости окон. ППР планируется за 12–18 месяцев, но его фактическое начало и длительность зависят от состояния действующего блока, обнаруженных дефектов, решений регулятора. Нередки случаи продления ремонта на недели. Если модель заложила «окно врезки» в определённый период ППР, а ремонт продлился и окно сдвинулось — весь план нового блока пересчитывается. Модель корректно описывает эту неопределённость через Monte Carlo, но не решает фундаментальную проблему: решение о выполнении врезки принимает главный инженер действующей станции, приоритет которого — безопасность действующего блока, а не график нового. Оптимизация одного без контроля над другим ограничена.

C-1-27
ID идеи: R1-1-22
Слабое место: Портфельная оптимизация нескольких проектов Росатома через единый метаграф с общими ресурсными узлами — идея стратегически ценная, но имеет организационный барьер. Оптимизация портфеля предполагает возможность централизованного перераспределения ресурсов между проектами — задержать один проект ради ускорения другого. В реальной структуре Росатома каждый проект имеет отдельный контракт с заказчиком (часто — иностранным государством), отдельного директора проекта, отдельный бюджет. Решение «забрать 50 сварщиков с проекта Аккую для проекта Эль-Дабаа» требует не оптимизации, а межгосударственных переговоров. Модель может показать оптимум, но механизм его реализации в действующей организационной структуре не существует.

C-1-28
ID идеи: R2-9-07
Слабое место: Байесовская оптимизация объёмов выборочного контроля качества с динамическим определением выборки — вступает в прямое противоречие с нормативной базой. Сокращение объёма контроля на основании «низкой априорной вероятности дефекта от надёжного подрядчика» не будет принято регулятором — Ростехнадзор требует выполнения нормативного объёма вне зависимости от статистической модели заказчика. Идея применима только к работам, где объём контроля не нормирован (общестроительные работы), но для них она избыточно сложна.

C-1-29
ID идеи: R2-5-10
Слабое место: Интеграция модели с требованиями кибербезопасности АСУ ТП — область с минимальной неопределённостью для графика. Требования кибербезопасности определяются нормативными документами (приказ ФСТЭК №239, требования Ростехнадзора) и включаются в проектную документацию. Мероприятия по кибербезопасности (изоляция сетей, установка средств защиты) — это определённый перечень работ с предсказуемой длительностью, которые встраиваются в график монтажа АСУ ТП. Вероятностная оптимизация здесь избыточна: неопределённость низка, объём работ определён проектом, последовательность — технологией. Это стандартная задача сетевого планирования, не требующая модели Дмитришина.

C-1-30
ID идеи: R2-4-02
Слабое место: Формирование оптимального портфеля типовых проектных решений для серии энергоблоков с помощью кластеризации и GNN — задача, хронологически не совпадающая с периодом сооружения конкретной АЭС. Решения о стандартизации и унификации принимаются на этапе разработки технического проекта (за 5–10 лет до строительства) и фиксируются в проектной документации. Кроме того, решения о стандартизации определяются не только стоимостью и сроками, но и стратегическими факторами (политика импортозамещения, отношения с ключевыми поставщиками, техническая политика корпорации), которые не поддаются формализации в GNN.

**C-4-01**
**ID идеи:** R2-1-01 (Стохастический анализ освоенного объёма)
**Слабое место:** Идея предполагает, что для каждой Монте-Карло реализации можно получить траектории PV, EV и AC с достаточной детализацией. На практике в существующих ИСУП (например, SPF/ИСУП) данные о фактической стоимости (AC) и освоенном объёме (EV) часто замыкаются постфактум, с задержкой в 1-2 месяца, и агрегируются по укрупнённым пакетам работ. Это делает практически невозможным построение тысяч вероятностных траекторий EV(t) и AC(t) с приемлемой точностью для оперативного управления. Модель столкнётся с проблемой «мусор на входе — мусор на выходе». Для её работы потребуется внедрение учёта затрат в реальном времени на уровне каждой работы, что является отдельной и крайне сложной организационно-технической задачей, выходящей далеко за рамки математической модели.

---

**C-4-02**
**ID идеи:** R2-1-02 (Оптимизация размещения тяжёлой техники)
**Слабое место:** Модель рассматривает краны как абстрактные «точки в пространстве с зоной охвата». В реальности даже сверхтяжёлые краны (Demag CC-8800) имеют ограничения не только по грузоподъёмности, но и по ветровым нагрузкам, необходимости периодического техобслуживания (которое может занимать недели) и потребности в специально подготовленных путях/площадках для перебазировки. Учёт всех этих факторов (ветер, обслуживание, подготовка путей) в многомерном пространстве атрибутов сделает модель вычислительно очень сложной. Без их учёта оптимизация может дать теоретически красивое, но практически невыполнимое расписание, когда кран «по плану» должен работать при штормовом предупреждении или перебазироваться без готового пути.

---

**C-4-03**
**ID идеи:** R2-1-03 (Оптимизация контрольных точек и инспекций регулятора)
**Слабое место:** Идея предполагает, что регулятор (Ростехнадзор или зарубежный аналог) будет подстраиваться под оптимальное расписание, сформированное моделью («заблаговременное бронирование инспекционных слотов»). В реальности регулятор — это внешняя, независимая сторона, чьи графики инспекций определяются его собственными регламентами, загрузкой и доступностью экспертов, а не оптимизационными потребностями проекта. Вероятность того, что регулятор согласится на предложенный моделью «оптимальный» слот, близка к нулю. Таким образом, оптимизация инспекций как внутреннего ресурса ошибочна. Более реалистично моделировать доступность инспекторов как экзогенный вероятностный процесс с очень жёсткими ограничениями, что снижает ценность оптимизации.

---

**C-4-04**
**ID идеи:** R2-1-07 (Многоагентное моделирование организационных взаимодействий)
**Слабое место:** Это крайне сложная научная задача. Определить функции полезности для каждого агента (Заказчик, Генподрядчик, Проектировщик и т.д.) и формализовать их «информационные задержки» в виде, пригодном для верификации и калибровки на реальных данных, практически невозможно. Результаты такой модели будут чувствительны к множеству субъективных допущений, что сделает её непригодной для принятия реальных управленческих решений. Проект рискует утонуть в настройке агентов, так и не получив практического инструмента. Модель может стать «игрушечной» и не отражать реальную сложность человеческих отношений и бюрократических процедур.

---

**C-4-05**
**ID идеи:** R2-2-04 (Атрибуция причин задержек для претензионной работы)
**Слабое место:** Идея возлагает на математическую модель задачу, которая по своей природе является юридической и контрактной. Определение «кто виноват» в задержке (concurrent delays) в строительстве — это сложный юридический анализ, опирающийся на условия контракта, переписку сторон, журналы работ. Свести это к формальным атрибутам на дугах графа — значит создать иллюзию объективности. На практике любая сторона, против которой будет свидетельствовать модель, найдет контраргументы, почему её «атрибут» был выставлен неверно или модель не учла форс-мажор. Вместо разрешения споров модель станет дополнительным полем для конфликта. Её выводы не будут иметь юридической силы без согласия сторон, которого, скорее всего, не будет.

**C-4-06**
**ID идеи:** R2-3-01 (Distributionally Robust Optimization)
**Слабое место:** DRO — мощный, но очень сложный для восприятия руководством и участниками проекта математический аппарат. Результаты такой оптимизации будет крайне трудно объяснить и защитить при принятии стратегических решений. «Мы выбрали эту стратегию, потому что она робастна к наихудшему распределению в рамках ambiguity set» — это не тот аргумент, который сработает на Совете директоров или при переговорах с заказчиком. Модель должна быть не только точной, но и интерпретируемой. Внедрение DRO рискует создать «чёрный ящик», решениям которого никто не будет доверять, несмотря на математическую красоту.

---

**C-4-07**
**ID идеи:** R2-6-02 (Инвертированная оптимизация под лимиты кредитных траншей)
**Слабое место:** Идея пытается решить финансовую проблему (лимиты годовых траншей) математическими методами управления расписанием. Но «растяжение» субкритических путей имеет естественные пределы, определяемые их свободным и полным резервами. Если финансовый лимит жёстче, чем эти резервы, то «вписать» затраты в транш без сдвига финального срока будет математически невозможно. Модель в таком случае просто выдаст ошибку или нереалистичное требование пересмотреть финансовые условия. Реальным решением было бы не «растяжение» графика, а пересмотр условий финансирования (увеличение транша, перенос части затрат на следующий период), что лежит вне компетенции модели управления расписанием.

---

**C-4-08**
**ID идеи:** R2-8-05 (Сентимент-анализ СМИ для GERT)
**Слабое место:** Идея опирается на очень шаткую корреляцию между сентиментом в соцсетях/СМИ и реальными действиями регулятора или общественности (судами, протестами, отказами в лицензиях). Сентимент может быть искусственно накручен, может не отражать настроения реальных активистов, имеющих право на судебные иски. Использование этого индикатора для изменения вероятностей в GERT-модели внесёт огромный «шум» и сделает прогнозы модели крайне волатильными и ненадёжными. Решения, принятые на основе «негативного твита», могут привести к необоснованным затратам на пиар и успокоение общественности там, где это не требуется, или, наоборот, к игнорированию реальной угрозы, которая не была должным образом освещена в сети.

## Раунд 3 — Критика

**C-6-01**
ID идеи: R1-1-18
Слабое место: Скрытое допущение о полноте данных BIM/ТИМ-модели для автоматического построения технологического графа (DAG). BIM-модель описывает конечную геометрию и объемы материалов, но она не содержит производственной логики: ограничений по доступности ресурсов, технологических перерывов (например, набор прочности бетона), пространственных зон безопасности и методов монтажа (например, укрупнительная сборка на СМБ или метод "Open Top"). Автоматическая трансляция BIM-элементов в сетевой график без массивного промежуточного слоя семантических «правил сборки» приведет к созданию астрономически огромного графа с избыточным и физически нереализуемым распараллеливанием работ. На таком графе симуляции Монте-Карло станут вычислительно невозможными из-за «проклятия размерности».

**C-6-02**
ID идеи: R2-1-07
Слабое место: Непреодолимая проблема параметризации и верификации многоагентной модели (MAS). Для корректной работы MAS требуются точные функции полезности, информационные горизонты и времена реакций для каждого участника (субподрядчики, заказчик, регулятор). В реальности эти параметры скрыты (из-за асимметрии информации), подвержены неформальным договоренностям и внешним политическим факторам. Откалибровать такую модель на исторических данных невозможно, так как истинные мотивы организационных задержек нигде официально не документируются. В результате модель будет выдавать прогнозы, зависящие исключительно от субъективных допущений разработчика (классический эффект "garbage in - garbage out").

**C-6-03**
ID идеи: R2-4-04
Слабое место: Критическая уязвимость алгоритмов обучения с подкреплением (Reinforcement Learning) к проблеме "sim-to-real gap" (разрыв между симуляцией и реальностью). Стройплощадка АЭС — это высокодинамичная и лишь частично наблюдаемая среда. На логистику влияют микрофакторы, не попадающие в цифровой двойник (локальная грязь на технологическом проезде, внезапная поломка стропов, усталость конкретной бригады). Агент RL будет генерировать диспетчерские решения, которые математически оптимальны в идеальной симуляции, но физически невыполнимы на реальной площадке. Кроме того, RL работает как "черный ящик", и линейный инженерный персонал не будет доверять неинтерпретируемым командам по переброске дефицитных ресурсов.

**C-6-04**
ID идеи: R2-8-05
Слабое место: Крайняя чувствительность к информационному шуму и манипуляциям. Использование сентимент-анализа соцсетей и СМИ для динамической корректировки вероятностей в строгой математической GERT-модели вносит дестабилизирующий фактор. Физические процессы должны быть предварительно просчитаны и аппроксимированы в виде легких суррогатных моделей, а не рассчитываться динамически внутри стохастического графа.

### C-9-01

**ID идеи:** R2-1-13  
**Автор:** Участник 1  
**Идея:** Применение Temporal Graph Networks (TGN) для прогнозирования «волн» каскадных задержек в графике строительства и оптимального момента превентивного вмешательства

**Критика:**  
Идея технически перспективна, однако содержит несколько слабых мест, требующих проработки.

1. **Проблема данных для обучения TGN.** TGN требуют временных рядов событий с высокой гранулярностью (минуты/часы). В строительстве АЭС фактические данные поступают с задержкой в дни/недели, а история похожих проектов ограничена. Как планируется решать проблему «холодного старта» модели на новом проекте?

2. **Неопределённость определения «волны».** В идее не формализован критерий, отличающий «критическую волну» от обычных флуктуаций. Без чёткого порога система будет генерировать ложные срабатывания, что приведёт к «alert fatigue» у диспетчеров.

3. **Игнорирование организационных факторов.** TGN моделирует технологические связи, но каскадные задержки часто возникают из-за организационных причин (задержка оплаты, согласований), которые не отражены в технологическом графе.

**Предложения по улучшению:**  
- Дополнить TGN многоагентной компонентой (R2-1-07) для учёта организационных задержек  
- Ввести двухуровневую систему: TGN для технологических волн + автоэнкодеры для детекции аномалий в организационных потоках  
- Разработать формальный критерий «силы волны» на основе отклонения от исторической дисперсии

### C-9-02

**ID идеи:** R2-2-01  
**Автор:** Участник 2  
**Идея:** Применить модель как контур управления системной и помещенческой передачей (turnover) от СМР к ПНР и последующей эксплуатации — не по «проценту стройготовности», а по вероятности готовности передаточных пакетов

**Критика:**  
Идея адресует важную проблему разрыва между логикой строителей и пусконаладчиков, но требует уточнений.

1. **Сложность определения «вероятности готовности».** В идее не раскрыт механизм расчёта вероятности готовности пакета. Если это субъективная экспертная оценка — модель теряет объективность. Если на основе ML — требуется обучающая выборка, которой пока нет.

2. **Конфликт интересов при передаче.** Строители заинтересованы в формальной передаче (чтобы закрыть фронт), пусконаладчики — в реальной готовности. Модель не учитывает этот конфликт и не предлагает механизма разрешения споров.

3. **Игнорирование регуляторных ограничений.** Регулятор (Ростехнадзор) устанавливает жёсткие требования к документации при передаче. Вероятностная модель может конфликтовать с детерминированными требованиями регулятора.

**Предложения по улучшению:**  
- Интегрировать байесовскую оценку «истинного факта» (R2-2-02) для объективизации вероятностей готовности  
- Включить в модель механизм арбитража при спорных передачах (например, автоматическая эскалация при расхождении оценок строителей и ПНР)  
- Добавить узлы GERT для регуляторных hold points с фиксированными длительностями согласований

### C-9-03

**ID идеи:** R2-3-04  
**Автор:** Участник 3  
**Идея:** Встраивание Physics-Informed Neural Networks (PINN) в дуги графа для моделирования физических процессов (набор прочности бетона, тепловые напряжения при сварке ГЦТ, остывание парогенераторов)

**Критика:**  
Идея амбициозна, но сталкивается с фундаментальными ограничениями.

1. **Вычислительная сложность.** PINN для каждой дуги графа потребуют решения дифференциальных уравнений в частных производных. При тысячах работ в графе АЭС это создаст неприемлемую вычислительную нагрузку на систему планирования.

2. **Несовместимость временных масштабов.** Физические процессы (набор прочности бетона — 28 дней, остывание ПГ — недели) существенно превышают шаг планирования СМР (дни). PINN будут давать предсказания с запаздыванием относительно оперативных решений.

3. **Проблема граничных условий.** Для PINN требуются точные граничные условия (температура, влажность), которые на стройплощадке измеряются с погрешностями и не всегда доступны в реальном времени.

**Предложения по улучшению:**  
- Использовать PINN не для всех дуг, а только для критических операций (бетонирование контейнмента, ТМР реакторного здания)  
- Заменить полноценные PINN на предобученные surrogate-модели (метамодели), сохраняющие физические ограничения, но работающие в реальном времени  
- Интегрировать с IoT-датчиками (R2-3-06) для автоматического сбора граничных условий

### C-9-04

**ID идеи:** R2-6-03  
**Автор:** Участник 6  
**Идея:** Автоматизированный математический фильтр заявок на изменения (Change Request) для защиты концепции «Design Freeze» с помощью графовых автоэнкодеров

**Критика:**  
Идея элегантна с математической точки зрения, но практическое применение сопряжено с рисками.

1. **Риск «ложных отрицаний».** Автоэнкодер может отклонить критически важное изменение (например, исправление ошибки проектирования), если оно существенно меняет топологию графа. Это создаёт риск сохранения дефектного проекта ради формального соблюдения «Design Freeze».

2. **Игнорирование бизнес-обоснований.** Модель оценивает только топологические изменения, но не учитывает экономическое обоснование (например, изменение может сократить сроки на 6 месяцев при умеренных затратах на пересогласование).

3. **Сложность интерпретации.** «Резкий рост ошибки реконструкции» — неинтерпретируемый критерий для принятия решения. Инженерам и менеджерам потребуется объяснение, почему конкретная заявка отклонена.

**Предложения по улучшению:**  
- Дополнить автоэнкодер SHAP-анализом для интерпретации «вклада» каждого изменения в ошибку реконструкции  
- Ввести двухуровневую систему: автоэнкодер — для выявления каскадных рисков, многокритериальная оптимизация — для финального решения с учётом бизнес-ценности  
- Добавить «белый список» типов изменений (критические исправления безопасности), которые не подлежат автоматическому отклонению

### C-9-05

**ID идеи:** R2-8-05  
**Автор:** Участник 8  
**Идея:** Сентимент-анализ локальных СМИ и социальных сетей для динамической корректировки вероятностей ветвления в узлах GERT-модели, связанных с общественными слушаниями, экологическими протестами и локальными разрешениями

**Критика:**  
Идея инновационна, но содержит методологические и практические проблемы.

1. **Проблема репрезентативности.** Социальные сети и СМИ отражают мнение активного меньшинства, а не «молчаливого большинства». Негативный сентимент в Twitter не коррелирует с реальным исходом слушаний напрямую.

2. **Задержка сигнала.** Сентимент-анализ даёт информацию постфактум (после публикации), тогда как для корректировки GERT требуется прогноз. Реактивная корректировка вероятностей не предотвратит кризис, а лишь зафиксирует его начало.

3. **Культурная специфика.** На разных рынках (Россия, Бангладеш, Турция, Египет) динамика общественного мнения существенно различается. Универсальная модель сентимент-анализа будет давать систематические ошибки.

4. **Риск манипуляций.** Конкуренты или заинтересованные группы могут организовать «накрутку» негативного сентимента для искусственного завышения вероятностей задержек в модели.

**Предложения по улучшению:**  
- Дополнить сентимент-анализ предиктивными индикаторами (например, активность НКО, судебные иски в аналогичных проектах, макроэкономические факторы региона)  
- Использовать критерий Вальда (R2-9-03) для выбора стратегии коммуникации, устойчивой к манипуляциям сентиментом  
- Ввести «фильтр достоверности» — корректировать вероятности только при подтверждении сигнала из независимых источников (соцсети + СМИ + активность НКО)
