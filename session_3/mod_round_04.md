## 6. Основные направления доработок

По итогам Раунда 4 прослеживаются устойчивые паттерны доработки:

**Сужение границ применения:** Участники ограничивают модель конкретными подзадачами вместо универсальных решений (RL → offline-советник; JIT → только для некритического оборудования; кибербезопасность → только график внедрения).

**Разделение жёстких и мягких ограничений:** Лицензии, бюджеты, доступность техники — как жёсткие constraints; вероятностная оптимизация — только внутри допустимого пространства.

**Снижение автономности модели:** От «автоматической системы» к «инструменту рекомендаций для человека». Human-in-the-loop как обязательный элемент.

**Учёт организационно-процедурных барьеров:** Формальные процедуры (переутверждение ПОС, согласования, наряды-допуски) включаются в граф как обязательные узлы с собственными длительностями.

**Федеративная архитектура данных:** Вместо единого датасета — федеративное обучение, изолированные экземпляры, иерархия моделей с локальными адаптерами.
