### Ключевые типы уязвимостей (повторяющиеся)

1. **Недостаточность данных для ML** — малое количество завершённых проектов, несопоставимость данных, отсутствие стандартизированных датасетов
2. **Разрыв модель ↔ организационная реальность** — алгоритмические рекомендации требуют контрактных согласований, переутверждения документации, регуляторных процедур
3. **Формализация неквантифицируемых процессов** — политика, социальные протесты, геополитика не имеют статистической базы
4. **Вычислительная сложность** — интеграция BIM, GNN/TGN на больших графах, непрерывный Монте-Карло
5. **Расфокусировка модели** — применение тяжёлого аппарата к задачам вне критического пути
6. **Ложная точность** — необоснованные числовые заявления (85–95%, 4–8 месяцев)

### Направления доработок

По итогам Раунда 4 выделяются пять устойчивых паттернов:

**Сужение границ применения.** Универсальные идеи конкретизированы до подзадач: RL → offline-советник; JIT → только для некритического оборудования; кибербезопасность → только график внедрения мер защиты; управление отходами → только радиоактивные.

**Разделение жёстких и мягких ограничений.** Лицензии, бюджетные лимиты, доступность уникального оборудования — как неустранимые constraints перед оптимизацией. Вероятностная оптимизация — только внутри допустимого пространства.

**Human-in-the-loop.** От «автоматических систем» к «инструментам рекомендаций». Модель предлагает сценарии, решение принимает руководитель с документированием обоснования.

**Процедурный слой в графе.** Формальные процедуры (переутверждение ПОС, согласования, наряды-допуски, инспекции) включены как обязательные узлы с собственными длительностями и вероятностями задержек.

**Федеративная архитектура.** Вместо единого корпоративного датасета — federated learning, изолированные экземпляры по юрисдикциям, иерархия моделей с локальными адаптерами. Решает вопрос информационной безопасности и domain shift.
